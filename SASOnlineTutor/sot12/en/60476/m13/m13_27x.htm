1.  

 Once you associate a libref with a DBMS, which of the following tasks can you perform?

a. update a SAS data set with DBMS data
b. combine DBMS data and SAS data
c. update DBMS data or create a DBMS table
d. all of the above


After you submit a SAS/ACCESS LIBNAME statement (or use the Explorer window) to associate a libref with a DBMS, you can perform any of the tasks listed above. Of course, you can also simply query the DBMS! 

2.  

 Suppose you submit the following SAS/ACCESS LIBNAME statement. How would you reference an Oracle table named Profits that is stored in the library? 
libname region1 oracle
        user=tester1 password=testpw
        path=ora7db;

a. Region1.Oracle
b. Oracle.Profits
c. Region1.Profits
d. Oracle.Region1


After associating a libref with a DBMS, you can reference the DBMS tables and views as if they were SAS data sets, using standard two-level SAS names. The first part of the name is the libref (Region1), and the second part is the name of the table (Profits). 


3.  

 Which SAS/ACCESS LIBNAME statement below associates the libref Mysybase with a SYBASE database and specifies engine connection options?

a. 
libname mysybase sybase user=testuser
        password=testpass database=testdb
        server=testserver;
b. 
libname sybase mysybase user=testuser
        password=testpass database=testdb
        server=testserver;
c. 
libname mysybase sybase dblink=sybase2
        preserve_col_names=yes;
d. none of the above


In the SAS/ACCESS LIBNAME statement, you specify the libref followed by the engine name, then any options. Engine connection options, such as USER= and PASSWORD=, are options that you specify to connect to a particular database. You can also specify additional options to control how your data is processed. 


4.  

 Which of the following statements is true?

a. Every DBMS supports all SAS/ACCESS LIBNAME options.
b. When you specify a SAS/ACCESS LIBNAME option, the option is applied to all objects (such as tables and views) in the database that the libref represents.
c. All SAS/ACCESS LIBNAME options are options that you specify to connect to a particular database.
d. all of the above


SAS/ACCESS LIBNAME options apply to the processing of objects and data in a DBMS, such as its tables or indexes. Support for many SAS/ACCESS LIBNAME options is DBMS specific.

6.  

 Which of the following options must you specify in order to create a DBMS table with column names that match your case-sensitive SAS variable names?

a. 
preserve_col_names=no
b. 
validvarname=any
c. 
preserve_tab_names=yes
d. 
preserve_col_names=yes


When you specify PRESERVE_COL_NAMES=YES as a SAS/ACCESS LIBNAME option, column names are read from and passed to the DBMS with special characters and exact spelling preserved. The maximum length for SAS data set names is 32 bytes.  


7.  

 Which of the following identifies the part of SAS that enables you to access your DBMS?

a. SAS/ACCESS LIBNAME options
b. SAS/ACCESS engine connection options
c. SAS/ACCESS-engine-name
d. all of the above


A SAS/ACCESS-engine-name is a SAS/ACCESS engine name for your DBMS, such as Oracle or DB2. SAS/ACCESS engines enable SAS to access DBMS data. SAS/ACCESS LIBNAME options are options that apply to the processing of objects and data in a DBMS, such as its tables or indexes. SAS/ACCESS engine connection options are options that you specify to connect to a particular database.  


8.  

 Which of the following sends dynamic, non-query, SQL statements to a DBMS?

a. EXECUTE statement
b. CONNECTION TO component (in the FROM clause of the PROC SQL SELECT statement)
c. CONNECT statement
d. DISCONNECT statement


The CONNECT statement connects to the DBMS. The CONNECTION TO component specifies the DBMS connection that you want to use. The EXECUTE statement sends DBMS-specific SQL statements to a DBMS for processing. The DISCONNECT statement terminates the connection to the DBMS. 


9.  

 What happens when you submit the following program? 
libname mysaslib 'c:\sas\sasdata';
libname myoracle oracle
        user=tester1 password=testpw
        path=ora7db;
libname myodbc odbc uid=testuser 
        pwd=testpass dsn=sqlservr;
proc sql;
   select employee.id, lastname, firstname,
          grosspay, accountbalance
      from mysaslib.employee, myoracle.payroll,
           myodbc.profitsharing
      where employee.id=payroll.id and
            payroll.id=profitsharing.id
      order by lastname;
quit;

a. The first LIBNAME statement generates errors because it doesn't specify a DBMS engine name and engine connection options. Then the PROC SQL step stops processing because the library Mysaslib isn't found.
b. The PROC SQL step joins SAS data, Oracle data, and Microsoft SQL Server data.
c. The PROC SQL step stops processing because it attempts to join incompatible data types.
d. The PROC SQL step stops processing because it contains syntax errors.


You can process data from multiple DBMS sources from within the same SAS program. Because the first LIBNAME statement references SAS data, it doesn't require an engine or engine connection options (in this case, SAS automatically chooses the appropriate engine). 


10.  

 How can you terminate the connection to a DBMS?

a. Submit a DISCONNECT statement in a PROC SQL step that connects to a DBMS.
b. Submit a SAS/ACCESS LIBNAME statement with the CLEAR option.
c. Without specifying a DISCONNECT statement, use a QUIT statement to end a PROC SQL step that connects to a DBMS.
d. all of the above


When you submit a SAS/ACCESS LIBNAME statement with the CLEAR option, the SAS/ACCESS engine disconnects from the DBMS and frees any resources that are associated with the connection. The DISCONNECT statement terminates the connection to the DBMS. If you omit the DISCONNECT statement, an implicit DISCONNECT is performed when PROC SQL terminates.
